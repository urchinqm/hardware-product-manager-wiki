# CAN

## 1. 基本概念

CAN（Controller Area Network，控制器局域网，是一种面向车辆和工业控制的多主通信总线）适合抗干扰、实时性和多节点通信场景。

CAN ID（CAN Identifier，CAN 报文标识符，用于表示报文优先级和含义）是 CAN 通信定义的核心。

## 2. 工作原理

CAN 节点通过差分信号发送报文，总线仲裁由 CAN ID 决定。ID 数值越小，仲裁优先级越高。节点通过 CRC（Cyclic Redundancy Check，循环冗余校验）和错误计数机制检测通信异常。

## 3. 核心参数

核心参数包括波特率、标准帧或扩展帧、CAN ID、DLC（Data Length Code，数据长度码）、报文周期、超时时间、终端电阻、线束长度、收发器型号和错误处理策略。

CAN FD（CAN Flexible Data-Rate，支持更大数据载荷和可变数据阶段速率的 CAN 扩展）需要通信双方和网络共同支持。

## 4. 参数影响

波特率影响总线长度和实时性。报文周期影响总线负载。终端电阻错误会造成波形反射和通信不稳定。ID 分配不合理会造成优先级冲突。

CAN vs UART 的差异是 CAN 支持多节点、仲裁和错误处理，UART 更简单但通常点对点。

## 5. 优点

CAN 抗干扰能力强，适合车辆和工业现场，具备仲裁和错误检测机制。

## 6. 局限性

CAN 载荷有限，协议设计和总线负载管理需要严格规划。

## 7. 产品经理关注点

产品经理必须把信号定义、报文周期、超时策略、诊断需求和兼容车型或设备边界写清楚。

## 8. 常见应用场景

常见于汽车 ECU（Electronic Control Unit，电子控制单元）、电池管理系统、车载配件、工业控制器、电机控制和仪表通信。

## 9. 需求文档需要明确内容

PRD 需要明确 DBC（Database CAN，CAN 信号数据库文件）来源、报文 ID、信号缩放、发送周期、接收超时、诊断码、升级影响和兼容测试矩阵。

## 10. 异常状态设计

异常状态包括 Bus Off（总线关闭）、错误帧过多、报文超时、ID 冲突、终端电阻异常、线束断开、信号值越界和版本不兼容。
