# OTA

## 1. 基本概念

OTA（Over-The-Air，空中升级，通过网络远程下发并安装软件或固件更新的能力）用于修复问题、发布新功能和安全补丁。

Firmware OTA（固件空中升级）和 App OTA（应用空中升级）需要区分升级对象和失败影响。

## 2. 工作原理

OTA 通常包括版本检测、升级包下载、完整性校验、签名校验、安装、重启、版本确认和失败回滚。

## 3. 核心参数

核心参数包括升级包大小、下载方式、校验算法、签名机制、断点续传、A/B 分区、回滚策略、最低电量、网络条件和灰度发布。

## 4. 参数影响

升级包越大，下载时间和失败概率越高。无签名校验会增加安全风险。无回滚策略会放大升级失败影响。

## 5. 优点

OTA 可远程修复问题，减少售后成本，并支持产品持续迭代。

## 6. 局限性

OTA 涉及设备端、App、云端和网络，异常路径复杂，必须设计完整。

## 7. 产品经理关注点

产品经理必须定义谁能升级、何时升级、能否跳过、失败如何恢复、用户是否感知和数据是否保留。

## 8. 常见应用场景

常见于固件升级、系统升级、App 更新、配置下发、算法模型更新和安全补丁发布。

## 9. 需求文档需要明确内容

PRD 需要明确版本规则、升级入口、强制升级、灰度策略、下载条件、安装时机、回滚、提示文案和日志上报。

## 10. 异常状态设计

异常状态包括无网络、下载失败、校验失败、空间不足、电量不足、安装失败、重启失败、回滚失败和版本不兼容。
