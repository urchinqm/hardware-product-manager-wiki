# Requirement Template

## 1. 基本概念

Requirement Template（需求模板，用于统一描述产品目标、功能范围、约束和验收标准的文档结构）能降低沟通偏差。

PRD（Product Requirements Document，产品需求文档）应说明要做什么、为什么做、做到什么程度以及如何验收。

## 2. 工作原理

需求模板通过固定章节约束信息输入，让背景、用户、场景、流程、状态、异常、数据、权限和测试标准被系统化描述。

## 3. 核心参数

核心字段包括目标、范围、用户场景、功能流程、状态机、接口、数据、权限、异常、埋点、验收标准和版本记录。

## 4. 参数影响

目标不清会导致方案漂移。范围不清会导致延期。验收标准不清会导致测试和研发理解不一致。异常缺失会导致线上问题。

## 5. 优点

模板能统一团队语言，提高评审效率，并减少遗漏。

## 6. 局限性

模板不能替代业务判断；照填模板但不澄清关键约束，仍然会产生低质量需求。

## 7. 产品经理关注点

产品经理应把需求写成可设计、可开发、可测试、可运营的输入，而不是愿望清单。

## 8. 常见应用场景

适用于新功能、硬件选型、App 功能、云端接口、设备联动、OTA、账号体系和项目立项。

## 9. 需求文档需要明确内容

模板应包含：背景、目标、术语、用户场景、功能范围、流程图、状态机、异常处理、数据定义、权限、埋点、验收标准、风险和版本记录。

## 10. 异常状态设计

需求模板必须要求每个核心流程列出失败、超时、取消、重试、降级、回滚和用户提示。
