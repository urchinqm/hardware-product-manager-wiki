# Exception Design Template

## 1. 基本概念

Exception Design（异常设计，定义系统在失败、边界和不可用情况下如何处理的需求设计）决定产品稳定性和售后成本。

Error Code（错误码，用于标识异常类型的编码）应服务于用户提示、日志定位和售后诊断。

## 2. 工作原理

异常设计先识别失败来源，再定义检测方式、用户提示、自动恢复、重试、降级、日志和最终兜底方案。

## 3. 核心参数

核心参数包括异常类型、触发条件、检测方法、影响范围、用户提示、自动恢复、重试次数、日志字段和售后处理。

## 4. 参数影响

异常提示不清会增加用户困惑。无日志会增加定位成本。重试策略不当会导致功耗、流量或服务器压力上升。

## 5. 优点

异常设计能提前暴露风险，使研发、测试、客服和售后对失败路径有统一处理方式。

## 6. 局限性

异常设计无法覆盖所有未知问题，必须配合日志、监控和版本迭代。

## 7. 产品经理关注点

产品经理要避免只写正常流程。每个关键动作都应有失败、超时、取消和恢复策略。

## 8. 常见应用场景

适用于网络、绑定、登录、OTA、支付、存储、传感器、权限、设备离线和云端服务异常。

## 9. 需求文档需要明确内容

PRD 需要明确异常列表、错误码、用户提示、重试策略、降级策略、日志字段、告警条件和售后指引。

## 10. 异常状态设计

异常状态应至少包含：可自动恢复、需用户操作、需售后介入、可忽略、可重试、不可重试和数据风险状态。
