# Test Case Template

## 1. 基本概念

Test Case（测试用例，用于验证需求是否被正确实现的测试条目）应覆盖正常流程、边界条件和异常路径。

Acceptance Criteria（验收标准，判断需求是否完成的明确条件）是测试用例设计的输入。

## 2. 工作原理

测试用例将需求拆成前置条件、操作步骤、输入数据、预期结果、异常结果和通过标准。

## 3. 核心参数

核心字段包括用例编号、需求来源、优先级、前置条件、步骤、测试数据、预期结果、实际结果、状态和备注。

## 4. 参数影响

没有需求追踪会导致漏测。预期结果不明确会导致争议。缺少异常用例会导致线上问题。

## 5. 优点

测试用例能让需求、研发和测试对验收标准形成一致理解。

## 6. 局限性

测试用例不能替代真实场景验证，硬件产品还需要环境、老化、兼容性和边界测试。

## 7. 产品经理关注点

产品经理应参与核心用例评审，尤其是用户路径、异常提示、状态机和售后高风险场景。

## 8. 常见应用场景

适用于功能测试、接口测试、兼容性测试、压力测试、功耗测试、OTA 测试和量产测试。

## 9. 需求文档需要明确内容

PRD 需要提供可测试的输入输出、状态变化、错误码、边界条件、性能指标和验收标准。

## 10. 异常状态设计

异常测试应覆盖网络中断、断电、低电量、空间不足、权限拒绝、设备离线、重复操作和版本不兼容。
