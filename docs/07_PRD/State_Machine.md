# State Machine Template

## 1. 基本概念

State Machine（状态机，用状态、事件和转移规则描述系统行为的模型）适合描述设备连接、升级、绑定、登录和异常恢复流程。

State（状态，系统在某一时刻的稳定条件）和 Event（事件，触发状态变化的输入）必须分开定义。

## 2. 工作原理

状态机通过“当前状态 + 事件 + 条件 = 下一个状态 + 动作”描述系统行为，避免不同团队对流程理解不一致。

## 3. 核心参数

核心元素包括状态列表、初始状态、终止状态、事件列表、转移条件、动作、超时、重试次数和异常状态。

## 4. 参数影响

状态定义不完整会导致边界行为不可测。超时和重试缺失会导致卡死。终止状态不清会导致用户界面无法正确展示结果。

## 5. 优点

状态机能把复杂流程变成可评审、可开发和可测试的规则。

## 6. 局限性

状态机过细会增加维护成本；过粗又无法覆盖异常。粒度需要服务于开发和测试。

## 7. 产品经理关注点

产品经理需要重点检查每个状态是否有进入条件、退出条件、用户可见文案和失败路径。

## 8. 常见应用场景

适用于配网、绑定、OTA、登录、支付、设备连接、录像状态、云端同步和生产测试。

## 9. 需求文档需要明确内容

PRD 需要明确状态表、事件表、转移表、超时、重试、用户提示、日志字段和测试用例覆盖。

## 10. 异常状态设计

异常状态包括超时、取消、失败、重试中、回滚中、部分成功、状态不一致和未知状态恢复。
